## Step 0: 環境チェック（最初に必ず実行）

### 0-1. MCP接続確認
以下の2つのMCPツールが利用可能か確認する：
- meta-ads MCP: `get_insights` で軽量クエリを実行して接続確認する（※ `health_check` / `get_ad_accounts` は `business_management` 権限が必要なため使用禁止）
- mcp-gsheets MCP: `sheets_check_access` で確認する

もし利用できないツールがある場合、以下の情報を出力して停止する：
```
[ERROR] MCP接続エラー
- 利用不可ツール: （ツール名）
- エラー内容: （エラーメッセージ全文）
- 想定原因:
  - meta-ads: META_ACCESS_TOKEN, META_APP_SECRET, META_APP_ID の環境変数が未設定または期限切れ
  - mcp-gsheets: GOOGLE_SERVICE_ACCOUNT_KEY が未設定、またはサービスアカウントにスプレッドシートの編集権限がない
- 対処法:
  - .mcp.json の設定を確認
  - 環境変数が正しくセットされているか確認
  - meta-ads の場合: Meta for Developers でトークンを再生成
  - mcp-gsheets の場合: スプレッドシートの共有設定でサービスアカウントのメールアドレスが編集者になっているか確認
```

### 0-2. スプレッドシート接続確認
Google Sheets MCP でスプレッドシート（ID: 1hGmdrm7MLqCQINx0zSeWzXIil17yHIsykAFfUBstX94）にアクセスできるか確認する。
アクセスできない場合：
```
[ERROR] スプレッドシート接続エラー
- SPREADSHEET_ID: 1hGmdrm7MLqCQINx0zSeWzXIil17yHIsykAFfUBstX94
- エラー内容: （エラーメッセージ全文）
- 想定原因:
  - スプレッドシートが存在しない、またはIDが間違っている
  - サービスアカウント（firebase-adminsdk-zzoo6@ukanalytics-37597.iam.gserviceaccount.com）にスプレッドシートの編集権限がない
- 対処法: スプレッドシートの共有設定を確認
```

## Step 1: Metaパフォーマンスデータの取得とシート書き込み

### 1-1. 実績ログ_キャンペーン シートを確認
「実績ログ_キャンペーン」シートを読み込み、既存データの有無を確認する。

### 1-2. Meta MCPでキャンペーンデータを取得
Meta MCP で広告データを取得する。
- account_id: 環境変数 META_ACCOUNT を使用
- 取得項目: campaign_name, status, impressions, clicks, spend, conversions, ctr, cpa, cpc, cpm

**データ取得期間の判定:**
- シートが空（ヘッダーのみ）の場合 → 初回実行。直近30日間のデータを取得
- シートにデータがある場合 → 前日（昨日1日分）のデータを取得

取得失敗時：
```
[ERROR] Meta API キャンペーンデータ取得エラー
- account_id: （使用したID）
- 取得期間: （指定した期間）
- エラー内容: （エラーメッセージ全文）
- HTTPステータス: （あれば）
- 想定原因:
  - 429: レート制限。5分後にリトライ
  - 401/403: アクセストークンが期限切れまたは権限不足
  - 400: account_idが不正
- 対処法: Meta for Developers でトークンの状態を確認
```

### 1-3. 「実績ログ_キャンペーン」シートに書き込み
取得したキャンペーンデータを以下の列に書き込む：
- A列 キャンペーンID
- B列 キャンペーン名
- C列 METAステータス（ACTIVE/PAUSED等）
- D列 日付（YYYY-MM-DD形式）
- E列 impressions
- F列 clicks
- G列 spend
- H列 conversions
- I列 CTR（%）
- J列 CPA
- K列 CVR（%）
- L列 CPM
- M列 備考

### 1-4. Meta MCPでクリエイティブ（広告）データを取得
Meta MCP で広告単位のデータを取得する。
- 取得項目: ad_id, ad_name, campaign_name, adset_name, impressions, clicks, spend, conversions, ctr, cpa, cpc, cpm

**データ取得期間の判定:**
- 「実績ログ_クリエイティブ」シートが空（ヘッダーのみ）→ 初回。直近30日間
- シートにデータあり → 前日1日分

取得失敗時：
```
[ERROR] Meta API クリエイティブデータ取得エラー
- エラー内容: （エラーメッセージ全文）
- 想定原因と対処法: （上記キャンペーンと同様）
```

### 1-5. 「実績ログ_クリエイティブ」シートに書き込み
取得した広告データを以下の列に書き込む：
- A列 広告ID
- B列 広告名
- C列 キャンペーン名
- D列 広告セット名
- E列 日付（YYYY-MM-DD形式）
- F列 impressions
- G列 clicks
- H列 spend
- I列 conversions
- J列 CTR（%）
- K列 CPA
- L列 CVR（%）
- M列 CPM
- N列 備考

書き込み失敗時：
```
[ERROR] Google Sheets 書き込みエラー
- 対象シート: （シート名）
- 書き込み行数: （行数）
- エラー内容: （エラーメッセージ全文）
- 想定原因:
  - 権限エラー: サービスアカウントに編集権限がない
  - シート名不一致: シート名が正しくない（「実績ログ_キャンペーン」等）
  - 行数超過: スプレッドシートの行数上限に到達
- 対処法: スプレッドシートの共有設定とシート名を確認
- 書き込めなかったデータ: （JSONで全データを出力）
```

### 1-6. パフォーマンス分析
書き込んだデータ + 既存データを使って分析する：
- METAステータス = "ACTIVE" のキャンペーンのみ対象
- CTR上位3件・CPA下位3件のキャンペーン名を特定
- 直近7日間のクリエイティブで CTR/CVR が高い上位5件、低い下位3件を抽出

## Step 2: コンテキスト読み込み

4. 「クリエイティブ・マスタ」シートを読み込む
   - Step 1で特定した上位・下位の広告IDに対応する行を取得
   - 訴求軸・ターゲット属性・見出し・バナー構成を確認

5. 「ナレッジDB」シートを読み込む（全行）
   - 「成功要因（勝ち）」「失敗要因（負け）」「黄金ルール」を把握

6. 「承認・FBログ」シートを読み込む（直近20行）
   - 判定が「×」だった提案の訴求パターンを把握
   - 同じパターンのクリエイティブを再提案しないこと

各シートの読み込み失敗時：
```
[ERROR] シート読み込みエラー
- 対象シート: （シート名）
- エラー内容: （エラーメッセージ全文）
- 対処法: シート名が正確に一致しているか確認（全角・半角・スペースに注意）
```

## Step 3: A/B案の生成

以下の3種類を各2案（A案・B案）生成する。合計6案。

### 生成ルール
- ナレッジDBの「黄金ルール」を必ず参照し、勝ちパターンを軸にする
- 承認FBログで過去に「×」になった訴求パターンは避ける
- A案とB案は訴求軸を明確に変える（例：A=価格訴求、B=信頼訴求）
- 文字数制限を厳守する（見出し30文字以内、説明文90文字以内）

### ① 広告コピー（テキスト）
- 見出し（30文字以内）× A案・B案
- 説明文（90文字以内）× A案・B案
- メインテキスト（自由）× A案・B案
- 訴求軸の名称（例：限定感、信頼、価格）

### ② バナー構成案
- レイアウト構成（例：上部ロゴ＋中央数字＋下部CTA）
- 視覚的訴求の方針（例：人物写真背景、ミニマル白背景）
- CTA文言の案

### ③ 画像生成プロンプト（英語）
- Midjourney/DALL-E向けの英語プロンプト
- スタイル・構図・雰囲気を具体的に記述
- 150〜200語程度

### 初回実行時の特別ルール
- 承認FBログが空 → ナレッジDBのみを参照してベストエフォートで生成
- ナレッジDBも空 → 実績ログのCTR/CPA上位クリエイティブのパターンのみを根拠に生成
- 両方空 → 一般的な広告ベストプラクティスに基づき生成し、その旨を特徴欄に明記

## Step 4: シートへの書き込み

### 「クリエイティブ・マスタ」に追記
新規行として以下の形式で追記する：
- 広告ID: 空欄（人が記入）
- 広告名: 提案_YYYYMMDD_A（または_B）
- 種類: コピー / バナー / 画像プロンプト
- メイン訴求: 生成した訴求軸
- サブ訴求: 副次的な訴求軸
- ターゲット属性: 参照した上位クリエイティブのターゲット
- 見出し（30文字）: 生成テキスト
- 説明文（90文字）: 生成テキスト
- メインテキスト: 生成テキスト
- バナー構成/動画台本: 生成した構成案
- 特徴: 生成根拠の要約（例：先週CTR1位の訴求を応用）
- ステータス: 承認待ち
- 作成日: 実行日（YYYY-MM-DD形式）
- デザインURL: 空欄
- AI識別タグ: AI_GENERATED

### 「運用変更ログ」に追記
- 日付: 実行日（YYYY-MM-DD形式）
- 対象キャンペーン・広告セット: 分析対象のキャンペーン名
- 変更種別: クリエイティブ提案
- 変更内容: 生成したA/B案の概要
- 理由・仮説: なぜこの案を生成したかの根拠
- 学習状態: 空欄
- 結果_7日後: 空欄（人が後日記入）

### 「承認・FBログ」に追記
- 提案日: 実行日（YYYY-MM-DD形式）
- 提案タイトル: A/B案提案：〇〇キャンペーン向け
- 判定: 空欄（人が記入）
- FB詳細（理由）: 空欄（人が記入）
- 当時の状況: 分析サマリー（CTR/CPA状況を具体的な数値で記載）
- 決定アクション: 空欄（人が記入）

各シートの書き込み失敗時は上記のエラーフォーマットで出力し、書き込めなかったデータをJSONで全文出力すること。

## Step 5: ナレッジDBの更新

以下の条件を満たす場合にナレッジDBを更新する：

条件1: 承認FBログに「×」がついた行がある
  → 失敗要因として記録し「次回の黄金ルール」に反映

条件2: 承認FBログに「◯」がついた案が実際に配信され、
        実績ログで CTR または CVR が平均+20%以上だった
  → 成功要因として記録し「勝ち」列に追記

更新形式：
- 更新日: 実行日（YYYY-MM-DD形式）
- 訴求軸・要素: 該当訴求軸の名称
- 成功要因（勝ち）: 具体的な訴求パターン
- 失敗要因（負け）: 避けるべきパターン
- 次回の黄金ルール: 次回生成時に守るべきルール
- 適用条件: どのキャンペーン・ターゲットに適用するか

## 品質チェックリスト（書き込み前に必ず確認）

- [ ] 見出しは30文字以内か
- [ ] 説明文は90文字以内か
- [ ] A案とB案は訴求軸が明確に異なるか
- [ ] ナレッジDBの「失敗要因」に該当する表現を使っていないか
- [ ] 承認FBログで過去に「×」になったパターンを繰り返していないか
- [ ] AI識別タグ（AI_GENERATED）を忘れずに記入しているか
- [ ] 運用変更ログに生成根拠を記録したか
