# PROMPT.md — スケジュールトリガー用プロンプト

以下のプロンプトをClaude Codeのスケジュールトリガーに設定してください。

---

## トリガープロンプト

```
CLAUDE.mdの「実行手順」に従い、Step 1〜5を順番に実行してください。

## Step 1: パフォーマンスデータの取得・分析

1. Meta MCP で直近7日間の広告データを取得する
   - account_id: 環境変数 META_ACCOUNT を使用
   - 取得項目: impressions, clicks, spend, conversions, ctr, cpa, cvr, cpm

2. Google Sheets MCP で「実績ログ_キャンペーン」シート（SPREADSHEET_ID: 1hGmdrm7MLqCQINx0zSeWzXIil17yHIsykAFfUBstX94）を読み込む
   - METAステータス = "ACTIVE" の行のみ対象
   - CTR上位3件・CPA下位3件のキャンペーン名を特定

3. 「実績ログ_クリエイティブ」シートを読み込む
   - 直近7日間のデータで CTR/CVR が高い広告IDを上位5件抽出
   - 直近7日間のデータで CTR/CVR が低い広告IDを下位3件抽出

## Step 2: コンテキスト読み込み

4. 「クリエイティブ・マスタ」シートを読み込む
   - Step 1で特定した上位・下位の広告IDに対応する行を取得
   - 訴求軸・ターゲット属性・見出し・バナー構成を確認

5. 「ナレッジDB」シートを読み込む（全行）
   - 「成功要因（勝ち）」「失敗要因（負け）」「黄金ルール」を把握

6. 「承認・FBログ」シートを読み込む（直近20行）
   - 判定が「×」だった提案の訴求パターンを把握
   - 同じパターンのクリエイティブを再提案しないこと

## Step 3: A/B案の生成

以下の3種類を各2案（A案・B案）生成する。合計6案。

### 生成ルール
- ナレッジDBの「黄金ルール」を必ず参照し、勝ちパターンを軸にする
- 承認FBログで過去に「×」になった訴求パターンは避ける
- A案とB案は訴求軸を明確に変える（例：A=価格訴求、B=信頼訴求）
- 文字数制限を厳守する（見出し30文字以内、説明文90文字以内）

### ① 広告コピー（テキスト）
- 見出し（30文字以内）× A案・B案
- 説明文（90文字以内）× A案・B案
- メインテキスト（自由）× A案・B案
- 訴求軸の名称（例：限定感、信頼、価格）

### ② バナー構成案
- レイアウト構成（例：上部ロゴ＋中央数字＋下部CTA）
- 視覚的訴求の方針（例：人物写真背景、ミニマル白背景）
- CTA文言の案

### ③ 画像生成プロンプト（英語）
- Midjourney/DALL-E向けの英語プロンプト
- スタイル・構図・雰囲気を具体的に記述
- 150〜200語程度

## Step 4: シートへの書き込み

### 「クリエイティブ・マスタ」に追記
新規行として以下の形式で追記する：
- 広告ID: 空欄（人が記入）
- 広告名: 提案_YYYYMMDD_A（または_B）
- 種類: コピー / バナー / 画像プロンプト
- メイン訴求: 生成した訴求軸
- サブ訴求: 副次的な訴求軸
- ターゲット属性: 参照した上位クリエイティブのターゲット
- 見出し（30文字）: 生成テキスト
- 説明文（90文字）: 生成テキスト
- メインテキスト: 生成テキスト
- バナー構成/動画台本: 生成した構成案
- 特徴: 生成根拠の要約（例：先週CTR1位の訴求を応用）
- ステータス: 承認待ち
- 作成日: 実行日
- デザインURL: 空欄
- AI識別タグ: AI_GENERATED

### 「運用変更ログ」に追記
- 日付: 実行日
- 対象キャンペーン・広告セット: 分析対象のキャンペーン名
- 変更種別: クリエイティブ提案
- 変更内容: 生成したA/B案の概要
- 理由・仮説: なぜこの案を生成したかの根拠
- 学習状態: 空欄
- 結果_7日後: 空欄（人が後日記入）

### 「承認・FBログ」に追記
- 提案日: 実行日
- 提案タイトル: 今週のA/B案提案：〇〇キャンペーン向け
- 判定: 空欄（人が記入）
- FB詳細（理由）: 空欄（人が記入）
- 当時の状況: 分析サマリー（CTR/CPA状況）
- 決定アクション: 空欄（人が記入）

## Step 5: ナレッジDBの更新

以下の条件を満たす場合にナレッジDBを更新する：

条件1: 承認FBログに「×」がついた行がある
  → 失敗要因として記録し「次回の黄金ルール」に反映

条件2: 承認FBログに「◯」がついた案が実際に配信され、
        実績ログで CTR または CVR が平均+20%以上だった
  → 成功要因として記録し「勝ち」列に追記

更新形式：
- 更新日: 実行日
- 訴求軸・要素: 該当訴求軸の名称
- 成功要因（勝ち）: 具体的な訴求パターン
- 失敗要因（負け）: 避けるべきパターン
- 次回の黄金ルール: 次回生成時に守るべきルール
- 適用条件: どのキャンペーン・ターゲットに適用するか

## 品質チェックリスト（書き込み前に必ず確認）

- [ ] 見出しは30文字以内か
- [ ] 説明文は90文字以内か
- [ ] A案とB案は訴求軸が明確に異なるか
- [ ] ナレッジDBの「失敗要因」に該当する表現を使っていないか
- [ ] 承認FBログで過去に「×」になったパターンを繰り返していないか
- [ ] AI識別タグ（AI_GENERATED）を忘れずに記入しているか
- [ ] 運用変更ログに生成根拠を記録したか

## エラー時の対応

- Meta APIが429エラー → 5分待ってリトライ。失敗なら運用変更ログにエラーを記録して終了
- Sheets書き込み失敗 → データをテキストで出力して通知
- 承認FBログが空（初回） → ナレッジDBのみを参照してベストエフォートで生成
- ナレッジDBが空（初回） → 実績ログのCTR/CPA上位クリエイティブのパターンのみを根拠に生成
```
